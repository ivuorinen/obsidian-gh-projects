# Obsidian GitHub Projects

Sync your GitHub repositories into Obsidian as markdown files. Each repo becomes a note with frontmatter metadata, open issues, and pull requests — designed to work with Obsidian's Bases feature for card-based project browsing.

## Features

- Fetches repositories via the GitHub GraphQL API in a single query
- Creates one markdown file per repo with structured frontmatter
- Lists open issues and pull requests with author, labels, and status
- Downloads repository cover images (social preview) for Bases card backgrounds
- Periodic auto-sync with configurable interval (15 min to 6 hours)
- Manual sync via command palette or ribbon icon
- Quick switcher to jump to any synced repo file
- Customizable output via a built-in template engine
- Secure token storage using Obsidian's SecretStorage API
- Configurable filters for forks, archived repos, and organization repos

## Installation

### BRAT (recommended for development)

1. Install the [BRAT plugin](https://github.com/TfTHacker/obsidian42-brat)
2. Open BRAT settings and select **Add Beta Plugin**
3. Enter `ivuorinen/obsidian-gh-projects`
4. Enable the plugin in Settings > Community Plugins

### Manual

1. Download `main.js`, `manifest.json`, and `styles.css` from the [latest release](https://github.com/ivuorinen/obsidian-gh-projects/releases)
2. Create a folder `obsidian-gh-projects` inside your vault's `.obsidian/plugins/` directory
3. Copy the three files into that folder
4. Enable the plugin in Settings > Community Plugins

## Setup

1. **Create a GitHub personal access token** at [github.com/settings/tokens](https://github.com/settings/tokens) with `repo` scope (for private repos) or `public_repo` scope (for public only)
2. Open the plugin settings in Obsidian
3. Under **Authentication**, click the GitHub token field to create or select a secret — your token is stored securely via Obsidian's SecretStorage and never written to disk
4. Enter your **GitHub username**
5. Optionally configure the output folder, assets folder, and sync interval
6. The plugin will sync automatically on startup, or use the command **"Sync GitHub repos now"**

## Settings

| Setting | Default | Description |
|---------|---------|-------------|
| GitHub token | — | Personal access token stored via SecretStorage |
| GitHub username | — | Username whose repos to fetch |
| Output folder | `GitHub` | Where repo markdown files are saved |
| Assets folder | `GitHub/assets` | Where cover images are downloaded |
| Template file | — | Optional template for custom body rendering |
| Sync interval | 60 min | Auto-sync frequency (15, 30, 60, or 360 min) |
| Issues limit | 25 | Open issues per repo (0, 10, 25, or 50) |
| PRs limit | 25 | Open PRs per repo (0, 10, 25, or 50) |
| Include forks | off | Include forked repositories |
| Include archived | off | Include archived repositories |
| Include org repos | off | Include organization repositories |

## Template System

By default, each repo file is rendered with a heading, description, and tables of issues/PRs. You can customize the body by pointing the **Template file** setting to a markdown file in your vault.

The plugin uses a `{{mustache}}` syntax for variable substitution. Frontmatter is always generated by the plugin (to ensure Bases compatibility) — only the body is customizable.

### Available Variables

| Variable | Description |
|----------|-------------|
| `{{repo.name}}` | Repository name |
| `{{repo.description}}` | Repository description |
| `{{repo.url}}` | GitHub URL |
| `{{repo.language}}` | Primary language |
| `{{repo.languages}}` | All languages (comma-separated) |
| `{{repo.topics}}` | Topics (comma-separated) |
| `{{repo.stars}}` | Star count |
| `{{repo.forks}}` | Fork count |
| `{{repo.watchers}}` | Watcher count |
| `{{repo.openIssues}}` | Open issue count |
| `{{repo.openPRs}}` | Open PR count |
| `{{repo.license}}` | License identifier |
| `{{repo.private}}` | Whether the repo is private |
| `{{repo.pushedAt}}` | Last push timestamp |
| `{{repo.updatedAt}}` | Last update timestamp |

### Block Iteration

Use `{{#issues}}...{{/issues}}` and `{{#pullRequests}}...{{/pullRequests}}` to iterate over items:

```markdown
{{#issues}}
- [#{{issue.number}}]({{issue.url}}) {{issue.title}} by @{{issue.author}} ({{issue.createdAt}})
{{/issues}}
```

**Issue variables:** `{{issue.title}}`, `{{issue.number}}`, `{{issue.url}}`, `{{issue.author}}`, `{{issue.labels}}`, `{{issue.createdAt}}`

**PR variables:** `{{pr.title}}`, `{{pr.number}}`, `{{pr.url}}`, `{{pr.author}}`, `{{pr.labels}}`, `{{pr.status}}`, `{{pr.createdAt}}`

See [`examples/github-repo-template.md`](examples/github-repo-template.md) for a complete example.

## Using with Bases

The plugin writes structured frontmatter to every repo file, making them work with Obsidian's Bases feature:

- Create a new Base and point it at your output folder
- Add properties like `stars`, `language`, `open_issues`, or `topics` as columns
- Set the `cover` property as the card background image
- Sort by `stars`, `updated_at`, or any other numeric/date field

The `cover` property is a vault-relative path to the downloaded social preview image, which Bases can render as a card background.
